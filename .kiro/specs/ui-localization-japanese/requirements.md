# Requirements Document

## Introduction

このドキュメントは、RTMSフロントエンドアプリケーションのUIテキストを統一された日本語に変更するための要件を定義します。現在のアプリケーションでは、一部のページが日本語化されている一方で、多くのページやコンポーネントが英語のままとなっており、ユーザーエクスペリエンスの一貫性が損なわれています。この移行により、日本のユーザーにとってより使いやすく、統一感のあるインターフェースを提供します。

## Glossary

- **UI Localization**: ユーザーインターフェースの言語とテキストを特定の地域や言語に適応させるプロセス
- **Technical Terms**: IP Address、MAC Address、JSONなど、一般的に英語で使用される技術用語
- **Business Terms**: システム固有の業務用語（デバイス、ユーザー、グループなど）
- **Action Labels**: ボタンやメニューのアクションを示すラベル（作成、編集、削除など）
- **Navigation Labels**: ナビゲーションメニューやパンくずリストで使用されるラベル
- **Form Labels**: フォームフィールドのラベルとプレースホルダー
- **Table Headers**: データテーブルのカラムヘッダー
- **Status Messages**: 成功、エラー、警告メッセージ

## Requirements

### Requirement 1

**User Story:** 開発者として、UIテキストの日本語化ガイドラインを確立したい。これにより、一貫した翻訳基準を適用できる。

#### Acceptance Criteria

1. WHEN 技術用語（IP Address、MAC Address、JSON、API、URLなど）が表示される THEN システムは英語表記を維持する
2. WHEN 業務用語（Device、User、Group、Product、Customerなど）が表示される THEN システムは適切な日本語（デバイス、ユーザー、グループ、製品、顧客）を使用する
3. WHEN アクション用語（Create、Edit、Delete、Save、Cancelなど）が表示される THEN システムは日本語（作成、編集、削除、保存、キャンセル）を使用する
4. WHEN ステータスメッセージが表示される THEN システムは自然な日本語を使用する
5. WHEN フォームバリデーションメッセージが表示される THEN システムは丁寧で分かりやすい日本語を使用する

### Requirement 2

**User Story:** ユーザーとして、ナビゲーションメニューが日本語で表示されることを望む。これにより、システムの機能を直感的に理解できる。

#### Acceptance Criteria

1. WHEN サイドバーナビゲーションが表示される THEN システムはすべてのメニュー項目を日本語で表示する
2. WHEN パンくずリストが表示される THEN システムは各階層を日本語で表示する
3. WHEN ページタイトルが表示される THEN システムは統一された日本語タイトルを使用する
4. WHEN ページの説明文が表示される THEN システムは分かりやすい日本語の説明を提供する
5. WHEN ナビゲーション要素にホバーした際 THEN システムは適切な日本語のツールチップを表示する

### Requirement 3

**User Story:** ユーザーとして、データテーブルのヘッダーとコンテンツが日本語で表示されることを望む。これにより、データの内容を素早く理解できる。

#### Acceptance Criteria

1. WHEN テーブルのカラムヘッダーが表示される THEN システムは適切な日本語ラベルを使用する
2. WHEN ソート機能を使用する THEN システムは日本語のソート指示を表示する
3. WHEN フィルタリング機能を使用する THEN システムは日本語のプレースホルダーと指示を表示する
4. WHEN ページネーション機能が表示される THEN システムは日本語のページ情報を表示する
5. WHEN 検索機能を使用する THEN システムは日本語の検索プレースホルダーを表示する

### Requirement 4

**User Story:** ユーザーとして、フォームのラベルと入力フィールドが日本語で表示されることを望む。これにより、正確にデータを入力できる。

#### Acceptance Criteria

1. WHEN フォームフィールドのラベルが表示される THEN システムは明確で分かりやすい日本語ラベルを使用する
2. WHEN 入力フィールドのプレースホルダーが表示される THEN システムは適切な日本語の例示を提供する
3. WHEN 必須フィールドの指示が表示される THEN システムは「必須」や「*」などの日本語指示を使用する
4. WHEN フィールドの説明が表示される THEN システムは分かりやすい日本語の説明を提供する
5. WHEN 選択肢（ドロップダウン、ラジオボタンなど）が表示される THEN システムは適切な日本語オプションを表示する

### Requirement 5

**User Story:** ユーザーとして、バリデーションエラーメッセージが日本語で表示されることを望む。これにより、エラーの原因を理解し、適切に修正できる。

#### Acceptance Criteria

1. WHEN 必須フィールドが未入力の場合 THEN システムは「[フィールド名]は必須です」形式のメッセージを表示する
2. WHEN 文字数制限に違反した場合 THEN システムは「[フィールド名]は[数字]文字以内で入力してください」形式のメッセージを表示する
3. WHEN フォーマットが不正な場合 THEN システムは「正しい[フォーマット名]を入力してください」形式のメッセージを表示する
4. WHEN 重複エラーが発生した場合 THEN システムは「この[フィールド名]は既に使用されています」形式のメッセージを表示する
5. WHEN 関連性エラーが発生した場合 THEN システムは分かりやすい日本語でエラーの内容を説明する

### Requirement 6

**User Story:** ユーザーとして、アクションボタンとダイアログが日本語で表示されることを望む。これにより、操作の意図を明確に理解できる。

#### Acceptance Criteria

1. WHEN アクションボタンが表示される THEN システムは「作成」「編集」「削除」「保存」「キャンセル」などの日本語ラベルを使用する
2. WHEN 確認ダイアログが表示される THEN システムは丁寧で分かりやすい日本語の確認メッセージを表示する
3. WHEN 削除確認ダイアログが表示される THEN システムは「この操作は取り消せません」などの適切な警告を日本語で表示する
4. WHEN 成功メッセージが表示される THEN システムは「正常に[アクション]しました」形式の日本語メッセージを表示する
5. WHEN ローディング状態が表示される THEN システムは「読み込み中...」「処理中...」などの日本語メッセージを表示する

### Requirement 7

**User Story:** ユーザーとして、ステータス表示とバッジが日本語で表示されることを望む。これにより、現在の状態を直感的に理解できる。

#### Acceptance Criteria

1. WHEN ユーザーロールが表示される THEN システムは「スーパーユーザー」「管理者」「一般ユーザー」の日本語ラベルを使用する
2. WHEN デバイスステータスが表示される THEN システムは「オンライン」「オフライン」「エラー」などの日本語ステータスを使用する
3. WHEN 処理ステータスが表示される THEN システムは「完了」「処理中」「失敗」などの日本語ステータスを使用する
4. WHEN 分類やカテゴリが表示される THEN システムは適切な日本語カテゴリ名を使用する
5. WHEN 数値や統計情報が表示される THEN システムは日本語の単位と説明を使用する

### Requirement 8

**User Story:** ユーザーとして、エラーメッセージとトースト通知が日本語で表示されることを望む。これにより、問題の内容を理解し、適切に対応できる。

#### Acceptance Criteria

1. WHEN API エラーが発生した場合 THEN システムは「データの取得に失敗しました」などの分かりやすい日本語エラーメッセージを表示する
2. WHEN ネットワークエラーが発生した場合 THEN システムは「接続に問題があります。しばらく待ってから再試行してください」などのメッセージを表示する
3. WHEN 権限エラーが発生した場合 THEN システムは「この操作を実行する権限がありません」などのメッセージを表示する
4. WHEN 成功通知が表示される THEN システムは「正常に保存されました」などの肯定的な日本語メッセージを表示する
5. WHEN 警告メッセージが表示される THEN システムは適切な注意喚起を日本語で表示する

### Requirement 9

**User Story:** ユーザーとして、日付・時刻表示が日本の形式で表示されることを望む。これにより、時間情報を自然に理解できる。

#### Acceptance Criteria

1. WHEN 日付が表示される THEN システムは「YYYY年MM月DD日」または「YYYY/MM/DD」形式を使用する
2. WHEN 時刻が表示される THEN システムは「HH:MM」または「HH:MM:SS」形式を使用する
3. WHEN 相対時間が表示される THEN システムは「○分前」「○時間前」「○日前」などの日本語表現を使用する
4. WHEN 期間が表示される THEN システムは「○日間」「○時間」などの日本語単位を使用する
5. WHEN タイムゾーン情報が必要な場合 THEN システムは「JST」または「日本標準時」を表示する

### Requirement 10

**User Story:** 開発者として、技術用語の英語表記を維持したい。これにより、国際的な標準に準拠し、技術者間のコミュニケーションを円滑にできる。

#### Acceptance Criteria

1. WHEN IP Address フィールドが表示される THEN システムは「IP Address」の英語表記を維持する
2. WHEN MAC Address フィールドが表示される THEN システムは「MAC Address」の英語表記を維持する
3. WHEN プロトコル名（HTTP、HTTPS、MQTT、TCP、UDPなど）が表示される THEN システムは英語表記を維持する
4. WHEN データ形式（JSON、XML、CSVなど）が表示される THEN システムは英語表記を維持する
5. WHEN ポート番号やURL関連の用語が表示される THEN システムは「Port」「URL」「API」などの英語表記を維持する

### Requirement 11

**User Story:** ユーザーとして、ヘルプテキストと説明文が日本語で表示されることを望む。これにより、機能の使い方を理解できる。

#### Acceptance Criteria

1. WHEN フィールドのヘルプテキストが表示される THEN システムは分かりやすい日本語の説明を提供する
2. WHEN 機能の説明が表示される THEN システムは簡潔で明確な日本語の説明を提供する
3. WHEN 操作手順が表示される THEN システムは段階的で理解しやすい日本語の手順を提供する
4. WHEN 注意事項が表示される THEN システムは重要なポイントを日本語で強調する
5. WHEN 例示が必要な場合 THEN システムは適切な日本語の例を提供する

### Requirement 12

**User Story:** 開発者として、一貫した翻訳パターンを適用したい。これにより、アプリケーション全体で統一された用語使用を実現できる。

#### Acceptance Criteria

1. WHEN 同じ概念を表す用語が複数箇所で使用される THEN システムは一貫した日本語用語を使用する
2. WHEN 階層的な関係を表す用語が使用される THEN システムは適切な日本語の階層表現を使用する
3. WHEN 動作を表す用語が使用される THEN システムは統一された日本語の動詞形を使用する
4. WHEN 状態を表す用語が使用される THEN システムは一貫した日本語の状態表現を使用する
5. WHEN 新しい機能が追加される THEN システムは既存の翻訳パターンに従った日本語用語を使用する
