# Skill: 日次設備解析レポート テンプレート

## 概要

日次設備解析レポートの構成・テンプレート・分析ルールを定義する。  
グラフ生成スクリプトは `chart_skill.md` を参照すること。  
PDFレポート生成は `pdf_report_skill.md` を参照すること。

---

## 対象

- **設備名：** 解析対象設備名を指定する
- **対象期間：** 当日 00:00:00 ～ 翌日 00:00:00 未満
- **対象データ：** 生産実績 / 稼働分類ログ / アラームログ

---

## 入力要件

### 必須入力

**生産実績**

| カラム | 代替名称例 |
|---|---|
| timestamp / datetime | started_at, ended_at, 時刻, 日時, 記録時刻 |
| 良品数 | good_qty, good_count, ok数 |
| 不良数 | ng_qty, ng_count, 不良数量 |

**稼働分類ログ**

| カラム | 代替名称例 |
|---|---|
| 開始時刻 | started_at, start_time, start |
| 終了時刻 | ended_at, end_time, end |
| 稼働分類 | status_name, group, operation_type |
| 継続時間 | duration, duration_min（なければ終了−開始で算出） |

稼働分類ログの `group` カラムは以下の4グループに分類される。

| group 値 | 意味 | 代表的な status_name |
|---|---|---|
| 操業時間 | 正常稼働 | 稼働 |
| 性能ロス時間 | 稼働しているが速度・品質上のロスあり | 速度低下、チョコ停 |
| 停止ロス時間 | 計画外停止 | 故障、段取り・調整、立上り、前干渉、後干渉、停止中 |
| 計画停止時間 | 計画的な非稼働 | 計画停止中、休憩時間、その他時間 |

**アラームログ**

| カラム | 代替名称例 |
|---|---|
| 発生時刻 | started_at, alarm_time |
| 復旧時刻 | ended_at（なければ継続時間で算出） |
| アラーム番号 | alarm_no, alarm_code |
| アラーム名称 | alarm_name, message |

### 任意入力

- 計画停止情報 / シフト情報 / 品種情報
- サイクルタイム / 設備条件 / 成形条件 / 品質測定値
- **基準サイクルタイム履歴（cycle_time_history）**：性能率・OEE 算出に必要
  - カラム：`applied_at`（JST）、`standard_cycle_time`（秒）
  - 対象日以前で最新の `applied_at` 時点の値を使用する

---

## 出力構成

出力は必ず以下の順で構成する。

1. 概要
2. 可視化（グラフ6種）
3. 分析結果
4. 考えられる要因
5. 追加確認事項
6. 参照データ
7. PDFレポート出力（`pdf_report_skill.md` の `build_pdf_report()` を呼び出す）

解析結果は簡潔に述べる。冗長な説明や断定的すぎる原因特定は避ける。

以下を必ず記載する。

- 対象設備
- 対象期間
- 取得データ件数（生産実績・稼働分類ログ・アラームログ）
- 主要カラム欠損の有無

**出力例**
```

```

### 3. 分析結果

#### 必須KPI表

| 指標 | 結果 |
|---|---:|
| 総生産数 | |
| 良品数 | |
| 不良数 | |
| 良品歩留（品質率） | |
| 計画稼働時間 | |
| 稼働時間 | |
| 停止ロス | |
| チョコ停（性能ロス） | |
| ロス合計 | |
| 可動率 | |
| 性能率 | |
| OEE | |
| アラーム件数 | |

#### 算出ルール

- 総生産数 = 良品数 ＋ 不良数（実績カラムが別にある場合は整合性確認）
- 良品歩留（品質率）= 良品数 ÷ 総生産数 × 100
- ロス合計 = 停止ロス ＋ チョコ停（性能ロス）
- 計画稼働時間 = 稼働分類ログのうち `group != "計画停止時間"` の区間合計（分）
  - すなわち「操業時間 ＋ 性能ロス時間 ＋ 停止ロス時間」の合計
- 可動率 = (計画稼働時間 − 停止ロス時間) ÷ 計画稼働時間 × 100
  - 停止ロス時間 = `group == "停止ロス時間"` の区間合計（分）
  - 正味稼働時間 = 計画稼働時間 − 停止ロス時間 = 操業時間 ＋ 性能ロス時間
- 性能率 = (総生産数 × 基準CT[秒] ÷ 60) ÷ 正味稼働時間[分] × 100
  - 基準CT = `cycle_time_history` の対象日時点で有効な `standard_cycle_time`（秒）
  - 基準CTが未提供の場合は性能率・OEE を「算出不可」とする
  - 性能率が 100% を超える場合は 100% に上限クリップする
- OEE = 可動率[%] × 性能率[%] × 品質率[%] ÷ 10000
- 時間単位は「分」
- 小数：時間は第1位、歩留・可動率・性能率・OEE は第1位まで

#### OEE 計算式（参考）

```
計画稼働時間 = Σ(group ≠ "計画停止時間") の区間長
正味稼働時間 = 計画稼働時間 − 停止ロス時間

可動率[%]  = 正味稼働時間 / 計画稼働時間 × 100
性能率[%]  = (総生産数 × 基準CT[秒] / 60) / 正味稼働時間[分] × 100
品質率[%]  = 良品数 / 総生産数 × 100

OEE[%]    = 可動率 × 性能率 × 品質率 / 10000
```

#### 文章要約ルール

以下の観点を3〜6点で簡潔にまとめる。

- 生産推移の概況
- 大きな落ち込み時間帯
- 長時間ゼロ生産帯の有無
- 最多アラームと発生時間帯
- 生産低下との関係示唆
- OEE の主要ボトルネック（可動率・性能率・品質率のうち最も低い指標）

---

### 4. 考えられる要因

断定せず、可能性ベースで記述する。

**記述観点**

- ロス構成上の主要因
- 最多アラームと設備部位の関係
- 生産低下時間帯との時間的一致
- OEE が低い場合、ボトルネック指標（可動率 / 性能率 / 品質率）の特定
- 追加データがないため断定不可である旨

**禁止事項**

- 根拠なく真因を断定しない
- 品質データがないのに品質要因を断定しない
- 保全履歴がないのに故障確定としない

---

### 5. 追加確認事項

必ず3件以上、優先度の高い順に列挙する。

---

### 6. 参照データ

以下を必ず明記する。

- データソース名
- 使用テーブル名（table_id）
- 対象設備名
- 対象期間
- 備考（ログ終端時刻・欠損状況など）

## 処理手順

### Step 1. データ抽出

対象日付の 00:00:00 から翌日 00:00:00 未満のデータを取得する。

### Step 2. データ前処理

- 時刻型を datetime に統一
- カラム名称の揺れを吸収（`find_col` 関数で対応）
- 重複レコードを確認
- 欠損値を確認
- 時系列ソート

### Step 3. 集計

- 30分単位で生産数（良品・不良）を集計
- 30分単位でアラーム件数を集計
- 稼働分類ごとの合計時間・件数を集計
- アラーム別件数・継続時間を集計
- KPIを算出（可動率・性能率・OEE を含む）

### Step 4. 異常兆候抽出

以下を優先して検出する。

- 生産数が急低下した時間帯
- 生産が0の継続時間帯
- アラーム集中時間帯
- 発生件数の多いアラーム
- 継続時間の長いアラーム

### Step 5. 可視化

`chart_skill.md` の6関数を呼び出す。

### Step 6. 要約生成

KPI表・箇条書き所見・考えられる要因・追加確認事項を簡潔にまとめる。

### Step 7. PDFレポート出力

`pdf_report_skill.md` の `build_pdf_report()` を呼び出してPDFを生成する。

```python
report_data = {
    "equipment_name": equipment_name,
    "target_date":    target_date,
    "kpi":            kpi_dict,         # Step 3で算出したKPI辞書（値は文字列）
    "findings":       findings_list,    # Step 6の所見リスト
    "factors":        factors_list,     # Step 6の要因リスト
    "action_items":   action_list,      # Step 6の追加確認事項リスト
    "data_note":      data_note_str,    # 参照データ・備考
}
chart_paths = [
    "production_alarm_trend.png",
    "operation_composition.png",
    "alarm_pareto.png",
    "oee_availability_dashboard.png",
    "cycle_time_trend.png",       # 存在しなければ自動スキップ
    "operation_timeline.png",
]
build_pdf_report(report_data, chart_paths,
                 output_path=f"{equipment_name}_{target_date}_report.pdf")
```

## 判定ルール

### 生産低下

以下のいずれかに該当した場合、低下として扱う。

- 30分値が当日中央値の70%未満
- 直前30分比で40%以上低下
- 0件

### アラーム集中

以下のいずれかに該当した場合、集中とみなす。

- 30分内件数が当日平均の2倍以上
- 同一アラームが短時間に連続発生

### 長時間停止候補

- 30分以上の連続ゼロ生産
- 稼働分類ログ上で停止系区分が継続
- アラーム復旧まで長時間を要したケース

### OEE 評価基準（参考）

| 評価 | OEE |
|---|---|
| 世界クラス | 85% 以上 |
| 良好 | 65〜85% |
| 改善余地あり | 65% 未満 |

## データ品質チェック

解析前に必ず以下を確認する。

- 対象日のデータが存在するか
- 時刻のタイムゾーンずれがないか
- 良品数・不良数が負値でないか
- 終了時刻が開始時刻より前になっていないか
- 同一アラームの重複記録がないか
- 生産実績の欠損時間帯が不自然でないか
- 稼働分類ログの区間が重複・欠落していないか（計画稼働時間の精度に直結）

問題がある場合は結果本文に短く注記する。

---

## 例外時のルール

| 状況 | 対応 |
|---|---|
| アラームログなし | 「アラームログなし」と記載。生産推移と稼働構成中心に報告 |
| 稼働分類ログなし | ロス分析・可動率・OEE は算出不可である旨を明記。生産実績とアラーム中心で報告 |
| 生産実績なし | 集計不能として明記。ログ取得異常または未稼働の可能性を記載 |
| データ一部欠損 | 可視化可能な範囲で出力。欠損・未取得を明記。原因の断定は避ける |
| 基準CTなし | 可動率のみ算出。性能率・OEE は「算出不可（基準CT未提供）」と記載 |

---

## 文体ルール

- 日本語で記述する
- 事実と推定を分ける
- 簡潔に書く（箇条書きを優先）
- 不要な比喩表現を使わない
- 断定表現を避け、「可能性がある」「考えられる」を用いる

## 成功条件

- 毎回同じ構成で出力される
- KPIが必ず算出される（基準CT不足時は可動率まで算出し、性能率・OEEは「算出不可」と明記）
- 可視化が最低5種類出る（Chart5はCTデータ取得できた場合のみ。基準CT未提供時はChart4の性能率・OEEを「算出不可」と表示して出力）
- 所見が簡潔である
- 根拠の薄い断定をしない
- データ欠損時も破綻せず報告できる
